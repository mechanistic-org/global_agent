# 🎨 UI Aesthetic Rules (MootMoat / Solstice)

When asked to style, build, or debug user interfaces, React/Astro components, or markdown structures across the MootMoat project, you MUST adhere to the following architectural visual constraints.

## 1. The Solstice Documentation Aesthetic
- **Constraint:** MootMoat is not a dark, unblinking SCADA control unit like Mechanistic. MootMoat is built on the Solstice Astro template, prioritizing accessible, high-legibility documentation and DIY clarity.
- **Execution:** Preserve the template's native typography, prose constraints, and clear hierarchies. Do not inject high-density monospace grids or severe monochromatic themes unless explicitly requested for a specific component.

## 2. Isomorphic Proofs
- **Constraint:** In the core colophon case studies for MootMoat, omit "Isomorphic Proof" blocks. These are intended exclusively for the Enriched `eriknorris` projects.
- **Execution:** Keep the MootMoat case studies focused strictly on the process, documentation, and the build itself.

## 3. The "Great Strip-Down" Principle
- **Constraint:** Avoid generic SaaS marketing copy and placeholder illustrations.
- **Execution:** When generating or modifying components, strip out "fluff." Focus on dense, functional documentation blocks, clear architecture diagrams, and utility over decoration.
